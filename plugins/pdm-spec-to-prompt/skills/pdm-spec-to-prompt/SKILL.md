# PdM｜仕様書 → 実装プロンプト スキル

仕様書を AI（Claude / Cursor / v0 / Lovable / Bolt 等）に渡す**実装プロンプト**に変換する。コードは書かない。「何を作るか」を曖昧さなく言語化することがゴール。

> **PdM-Playbook 上の位置**: Skill 4。前ステップは `pdm-design-doc`（Skill 3 で仕様書を作る）。実装側は `nextjs-mockup` などのコーディング系スキルへ引き渡す。

---

## まず確認すること（仕様書の必須要素）

仕様書に以下が揃っていないと、プロンプトが曖昧になり AI が暴走する。欠けている項目があれば `pdm-design-doc` に戻ってもらう：

- [ ] **概要** — 何のために何を作るか（1段落）
- [ ] **対象ユーザー** — ロール・利用シーン（「ログイン済みユーザー」では NG）
- [ ] **ハッピーパス** — 正常系のフロー
- [ ] **エッジケース** — 空・上限・エラー・権限
- [ ] **UI 状態** — デフォルト / ローディング / エラー / 空状態 / 成功
- [ ] **やらないこと** — スコープ外の明示
- [ ] **技術スタック** — 言語・FW・主要ライブラリ

---

## 4ステップの実行手順

### Step 1｜検証

上のチェックリストを順に確認する。仕様書を読みながら：

- 抜けている項目を**列挙**してユーザーに返す
- 「対象ユーザーがログイン済みユーザー」のように粗い場合は具体化を求める
- スタックが書かれていなければ「どの技術で作りますか？（例: Next.js + TypeScript + Tailwind）」と問い返す

### Step 2｜生成

以下のテンプレートで実装プロンプトを生成する：

```markdown
# 実装プロンプト｜[機能・施策名]

## ゴール（Goal）
**[ユーザー]** が **[シーン]** で **[達成したいこと]** をできるようにする。
これにより **[改善される指標 / 状態]**。

## 技術スタック（Tech Stack）
- **言語 / FW**: 例) Next.js 14（App Router）/ TypeScript strict
- **UI**: 例) Tailwind CSS / shadcn-ui
- **データ**: 例) Drizzle ORM / Supabase Postgres
- **その他**: 例) Server Actions / RLS / Vitest

## 要件（Requirements）

### 機能
- [機能1の概要]
  - ハッピーパス: ユーザーが〇〇する → システムが〇〇する
  - 入力: [フィールドと制約]
  - 出力: [画面・データの変化]

### エッジケース・例外処理
- 入力が空のとき: [挙動]
- 上限を超えたとき: [挙動]
- 権限がないとき: [挙動]
- ネットワーク切断時: [挙動]
- API エラー時: [挙動]

### UI 表示状態
- デフォルト: [何が見えるか]
- ローディング: [何を表示するか]
- 空状態（データなし）: [プレースホルダー / CTA]
- エラー: [メッセージとリカバリ手段]
- 成功: [完了表示・次の動線]

## 作らないもの（Out of Scope）
- [機能 X] — 理由: 今回のリリース外
- [機能 Y] — 理由: 別 PR で扱う

## 出力指定（Output）
- 生成するファイル一覧（パスと役割）
  - `app/feature/page.tsx` — エントリ画面
  - `app/feature/actions.ts` — Server Action
  - `components/feature/Form.tsx` — 入力フォーム
- コードに含めないもの: テストファイル / Storybook / コメント過多
- 既存コードへの変更: 〇〇のみ可 / 触らない

## 制約（Constraints）
- 既存の `lib/auth.ts` の認証関数を使うこと
- スタイルは Tailwind のみ。CSS Modules / styled-components 不可
- API キーやシークレットは `process.env.*` 経由
- 1コミット = 1機能の粒度
```

### Step 3｜品質チェック

生成後、以下を確認する：

| チェック項目 | OK / NG |
|---|---|
| Goal が「ユーザー × シーン × 達成」で書けている | |
| 技術スタックが具体的（バージョンや FW 名まで） | |
| エッジケースが網羅されている | |
| UI 状態が5種すべて定義されている | |
| Out of Scope が明示されている | |
| 出力指定にファイルパスが書かれている | |

NG があれば**仕様書側に戻して補完**する。プロンプト側で曖昧さを埋めない。

### Step 4｜納品

- プロンプトを Markdown ブロックでそのまま出力（コピペで AI ツールに渡せる形）
- 機能が大きい（5画面以上 / 複数 API）場合は**分割して納品**：
  - 「画面1: 一覧ページ」のプロンプト
  - 「画面2: 詳細ページ」のプロンプト
  - 「API: データ取得」のプロンプト
- どの順で実装するかを末尾に**実装順の推奨**として添える

---

## プロンプトを書くときの原則

| 原則 | 理由 |
|---|---|
| **具体的に書くほど良い** | 曖昧な指示は AI が想像で埋める。期待と違う実装が出る最大原因 |
| **作らないものを明示する** | AI は親切心で過剰に作りがち。Out of Scope で抑える |
| **既存コードへの言及** | 「既存の〇〇を使え」と書かないと AI は新規に作り直す |
| **状態 = 全部書く** | UI 状態は 5 種ある（デフォ・読込・空・エラー・成功）。1つでも欠けると詰む |
| **大きい機能は分割** | 1プロンプトに複数機能を詰めると AI の集中が割れる |

---

## ツール別の注意点（参考）

| ツール | 注意 |
|---|---|
| Claude Code | 既存ファイルを読ませてから渡す。CLAUDE.md / rules でガードレールを設ける |
| Cursor | `@file` でコンテキスト指定。Composer モードで複数ファイル編集 |
| v0 / Lovable / Bolt | UI を作らせるのが得意。バックエンドは別途プロンプト化 |
| GitHub Copilot Chat | 範囲が狭い。1プロンプト1機能で割る |

---

## よくある失敗パターン

| 失敗 | 対処 |
|---|---|
| 「いい感じに作って」と書いてしまう | 具体的な状態・データ・遷移を書く |
| 仕様書のコピペで終わる | 出力指定（ファイルパス・スタック）を必ず追加する |
| エッジケースが「適切にエラー処理」止まり | 「空のとき X を表示」と動作レベルで書く |
| Out of Scope が無い | AI は「ついでに」を作るので必ず明示する |
| 大きい機能を1プロンプトに詰める | 画面 / API / モデル単位に分割する |

---

## 重要な原則

- **PdM はプロンプトの品質保証者**。AI が出したプロンプトをそのまま渡さない
- **仕様書が曖昧ならプロンプトも曖昧になる**。仕様書に戻る勇気が必要
- **このスキルの出口**: コピペで AI コーディングツールに渡せるプロンプト。コードを書くのは別スキル

# Issue Intake (As-Is / To-Be → Issue) スキル

このスキルは、**Issue を起票するフェーズ** だけを担当する。PdM/CS や、思いついた要望をまずチケット化したい人が入口で使う。実装フェーズには踏み込まない。

---

## このスキルを使う場面

✅ **使う**:
- 「As-Is: ○○ / To-Be: ○○」と整理された入力
- 「○○できるようにしたい」「○○がバグってる」「○○あると嬉しい」などのふわっとした要望
- 「これチケットにして」「Issue 作って」「課題を起票したい」
- 複数の要望をまとめて起票したい(1 件ずつ繰り返し適用)

❌ **使わない**:
- 既に Issue 番号が決まっている → `github-issue-pr-flow`(実装フェーズ)へ
- アイデア段階で仕様自体を整理したい → `pdm-design-doc` へ
- 雑談・調査・読み取りのみ

---

## 上位/下位スキルとの関係

```
pdm-design-doc          (アイデア → Design Doc・仕様書)
       ↓
issue-intake            ← このスキル (As-Is/To-Be → Issue)
       ↓
github-issue-pr-flow    (Issue → ブランチ → 実装 → PR)
```

各スキルは **一方向に渡すだけ**。実装まで戻らない。

---

## 標準フロー

### Step 1. 入力を As-Is / To-Be に整理

入力パターンによって対応を分岐する。

#### パターン A: As-Is / To-Be が明示されている
そのまま使う。曖昧な部分があれば 1 文ずつ確認する。

#### パターン B: 不具合報告風 (「○○が壊れてる」「動かない」)
- As-Is = 現在の壊れた挙動
- To-Be = 正常に動く状態
- 種別ラベル = `bug` 候補

#### パターン C: 機能要望風 (「○○できたら嬉しい」「○○あるといいな」)
- As-Is = 「現状 ○○ できない / ○○ が無い」
- To-Be = 入力された要望
- 種別ラベル = `enhancement` 候補

#### パターン D: ふわっとしすぎ (「なんか UX 微妙」「使いづらい」)
- 推定せず、**ユーザーに質問** して具体化する
  - 「どの画面のどの操作が、どう微妙ですか?」
  - 「どうなったら満足ですか?」
- 具体化できなければ起票しない(`pdm-design-doc` への誘導を案内)

**整理後、必ずユーザーに確認** してから次へ進む:

```
以下の内容で起票しようとしています。よろしいですか?

As-Is: <要約>
To-Be: <要約>
推定ラベル: <種別ラベル>
```

ユーザー承認なしに Step 2 へ進まない。

---

### Step 2. 実際のコードを軽く確認し、対象箇所を特定する

**目的**: PdM が「この Issue を実装するか否か」を判断するための材料を Issue に盛り込む。実装方針の検討や影響範囲の網羅調査は **しない**(それは `github-issue-pr-flow` の役割)。

#### やること(軽量に留める)
- As-Is / To-Be のキーワード(画面名・機能名・UI 要素・エラーメッセージ等)で 1〜2 回 `grep` / ファイル名検索
- 関連しそうなファイル・コンポーネントを 1〜2 個メモする程度
- 重複チェック(Step 4)のキーワード精度を上げるための語彙(関数名・コンポーネント名など)を拾う

#### やらないこと
- 詳細な実装方針の検討
- 影響範囲の網羅調査
- 関数レベルの内部ロジック理解
- リファクタや「ついでに修正」の提案

#### 特定できた情報の使い方
- Step 4 の重複チェックで使うキーワードを補強する
- Step 5 の Issue 本文「補足」セクションに該当ファイル / コンポーネントを 1〜2 行記載
- ふわっとした要望でも「○○画面の○○ボタンあたり」と PdM が判断できる粒度に上げる

#### 特定できない / 候補が多すぎる場合
- 該当箇所が見つからない → As-Is が曖昧すぎる可能性が高い。Step 1 に戻ってユーザーに確認
- 候補が多数ヒットして絞れない → ユーザーに「どのあたりの話ですか?」と聞く
- それでも特定できなければ「対象箇所未特定」と明記して進む(PdM 判断に委ねる)

#### Issue 運用していない / コードリポジトリではない場合
- このステップは丸ごとスキップして Step 4 へ

---

### Step 3. Issue サイズの見積もり

**原則**: **1 Issue = 1 Value**(独立した価値の単位)を基本とする。**Value = ユーザーが体感できる変化** であり、内部実装の単位ではない。

**目的**: 起票フェーズで粒度を整え、レビュー負荷・実装計画の見通しを良くする。`size/XL` 相当なら起票前に分割を提案する。

#### サイズ目安(業界一般値・Google 推奨を叩き台)

| ラベル | 行数目安 | レビュー目安 | 典型例 |
|---|---|---|---|
| `size/S` | 〜100 行 | 15 分以内 | 文言修正・軽微なバグ・1 コンポーネント変更 |
| `size/M` | 100〜300 行 | 30〜60 分 | 1 機能の素直な実装(**推奨レンジ**) |
| `size/L` | 300〜500 行 | 1〜2 時間 | 大きめだが分割しにくい 1 機能。要相談 |
| `size/XL` | 500 行+ | — | **原則分割**。複数 Issue に分けるか、設計レビューを別途実施 |

#### やること

- Step 2 で特定した対象箇所をもとに、**想定 PR サイズと根拠**(変更ファイル数・新規/既存比率など)を 1〜2 行で提示する
- 提示フォーマット例:

```
想定サイズ: size/M(〜300 行・1 機能の素直な実装)
根拠: 新規 1 ファイル(コンポーネント) + 既存 2 ファイル軽微修正 / テストは別途
```

#### `size/XL` 相当の場合 → 分割提案

複数 Issue 案(各 Issue のタイトルと簡単な As-Is/To-Be)を提示し、ユーザー承認後に **それぞれ Step 1 から再実行**。

#### `size/S` 相当が連続して並ぶ場合 → 統合提案

近接 Issue の重複チェック前段として、**既存 open Issue があれば乗せる(コメント追記)** ことを提案する。Step 4 の選択肢 a)〜d) と接続する。

#### 分割の判断軸(エッジケースとも連動)

以下が **すべて Yes** なら 1 Issue としてまとめてよい。1 つでも No なら分割を検討:

- 1 つの **受け入れ条件** で完了を綺麗に書けるか
- ユーザー視点で **独立した価値** として完結するか(他の Issue の完了に依存しない)
- **動くものとしてのレビュー単位** になっているか(中途半端なリファクタ + 機能追加が混ざっていないか)

#### 推定できない場合

- 対象箇所が未特定(Step 2 で「対象箇所未特定」と判断)なら `size/L` を仮置きし、本文に「サイズ未確定。PdM 判断後に再見積もり」と明記
- 自信が無ければ広めに振る(S/M 迷うなら M)

---

### Step 4. 既存 Issue の重複チェック

`gh issue list --search "<キーワード>" --state all --limit 20` で検索する。

#### キーワードの作り方
- To-Be / As-Is の主要名詞・動詞を 2〜3 個抽出
- 同義語・関連語も試す(例: 「通知」+「アラート」+「メール」)
- 必要に応じて `--label <ラベル>` で絞り込み

#### 検索結果の提示フォーマット

```
重複チェック結果:

🔴 強い候補(open):
- #12 〇〇画面の通知改善 (created: 2026-04-01, labels: enhancement, P2)

🟡 弱い候補(closed):
- #23 〇〇通知をメールで送る (created: 2026-03-10, closed: 2026-03-15)

選択肢:
a) #12 に乗せる(コメントで追記情報を投稿)
b) 新規 Issue を起票
c) #23 を再オープンして使う
d) やめる
```

**ユーザーが選ぶまで起票しない**。

#### 何件もヒットして判定できない場合
- ラベルや期間で絞り込む提案をする
- それでも絞れなければユーザーに「直近 1 ヶ月のみ見ますか?」「特定ラベル無しで?」と確認

---

### Step 5. Issue 起票(b を選んだ場合のみ)

#### 5-a. ラベル確認

既存ラベル一覧を取得:

```bash
gh label list --limit 100
```

**種別ラベルは必須**。To-Be の内容から推定:
- バグ → `bug`
- 新機能・改善 → `enhancement`
- ドキュメント → `documentation`
- 内部リファクタ → `refactor` または `tech-debt`
- セキュリティ → `security`
- テスト追加 → `test`

**サイズラベルも必須**(種別と並列)。Step 3 の見積もりに基づき、以下のいずれかを付ける:
- `size/S` / `size/M` / `size/L` / `size/XL`

該当ラベル(種別 or サイズ)が無い場合は **ユーザーに確認**:「`X` を付けようと思いますが、新規ラベル `Y` を作りますか?」承認なしに `gh label create` しない。サイズラベル群がリポジトリに無ければ、初回起票時にまとめて作成提案する(承認後 `gh label create size/S` 等)。

優先度・領域ラベル(`P0`/`P1`/`ux`/`manual` 等)はユーザーが明示したときだけ。

#### 5-b. タイトル

- 70 文字以内
- 「○○を△△できるようにする」「○○のときに○○を表示する」のような **To-Be 起点** の動詞文
- リポジトリの既存 Issue のスタイルに合わせる(`gh issue list --limit 10` で確認)
- プレフィックス例: `feat:` `fix:` `docs:` `refactor:` `chore:` (リポジトリの慣習に合わせる)

#### 5-c. 本文テンプレート

```markdown
## As-Is
<現状を 1〜3 文で>

## To-Be
<あるべき姿を 1〜3 文で>

## 想定サイズ
size/M(〜300 行・1 機能の素直な実装)
根拠: <変更想定ファイル・新規/既存の比率など 1〜2 行>

## 受け入れ条件
- [ ] <これを満たせば完了とみなせる条件>
- [ ] <なるべく検証可能な粒度で>
- [ ] (UI 変更を伴う場合) マニュアル / リリースノート / スクショも同じ PR で更新
- [ ] (該当する場合) E2E テストの追加・更新

## 補足
<関連 Issue / 関連 URL / スクショ / 参考実装 / Step 2 で特定した該当ファイル・コンポーネント / Step 3 のサイズ根拠の詳細 等。空ならセクション削除>
```

**受け入れ条件は強く推奨**。「実装フェーズで何を作れば完了か」を入口で固めることで、後工程の認識ズレを防ぐ。書けない場合は「ユーザー側で後ほど補足してください」と明記して起票して良い。

**マニュアル / リリースノート / E2E** の項目は、プロジェクトに該当機能が無ければ自然と省略される(チェックボックスごと外して起票)。プロジェクトの `CLAUDE.md` にマニュアル/リリースノートの運用が書かれている場合は、それを参照する旨を補足に入れると親切。

#### 5-d. `gh issue create` 実行

```bash
gh issue create \
  --title "<タイトル>" \
  --label "<種別ラベル>" \
  --label "<追加ラベル(任意)>" \
  --body "$(cat <<'EOF'
<本文テンプレート>
EOF
)"
```

#### 5-e. ラベル付与の確認

```bash
gh issue view <番号> --json labels
```

ラベル空配列ならエラー。`gh issue edit` で追加してから次へ進む。

---

### Step 6. 既存 Issue にコメント追加(a を選んだ場合のみ)

```bash
gh issue comment <番号> --body "$(cat <<'EOF'
追加情報:

## As-Is
...

## To-Be
...

(関連: 今回 issue-intake skill 経由で追記)
EOF
)"
```

ラベルや優先度が変わるべきなら、ユーザーに確認してから `gh issue edit` で更新する。

---

### Step 7. 再オープン(c を選んだ場合のみ)

```bash
gh issue reopen <番号>
```

その上で Step 6 と同じくコメント追記。再オープンの理由が「クローズ判断が早すぎた」のか「再発した」のかを 1 文でコメントに含める。

---

### Step 8. 完了報告

ユーザーに以下を報告して **作業を止める**:

- Issue URL
- 付けたラベル
- ステータス(新規起票 / コメント追記 / 再オープン)
- **「実装は `github-issue-pr-flow` に `#<番号>` を渡してください」と引き継ぎ案内**

**禁止事項**:

🚫 そのままブランチを切らない
🚫 そのまま実装に進まない
🚫 「ついでに直しました」をしない
🚫 ユーザー承認なしに新規ラベルを作らない

実装まで進めたい場合はユーザーが明示的に「これ実装して」と次の入力をする。その時点で `github-issue-pr-flow` の役割。

## エッジケース

### 入力が複数の課題を含んでいる / 大きすぎる Issue が来た場合
- 1 課題ずつ分解してから Step 1 に戻る
- 「3 件に分かれそうですが、それぞれ起票しますか? まとめますか?」とユーザー確認
- Step 3 で `size/XL` 相当と判定された場合も同様に分割提案

#### 分割の判断軸

以下が **すべて Yes** なら 1 Issue としてまとめてよい。1 つでも No なら分割を検討する:

- 1 つの **受け入れ条件** で完了が綺麗に書けるか
- ユーザー視点で **独立した価値** として完結するか(他の Issue の完了に依存しない)
- **動くものとしてのレビュー単位** になっているか(中途半端なリファクタ + 機能追加が混ざっていない)

逆に **統合の判断軸**: 連続する複数の `size/S` 相当が以下を満たすなら 1 Issue にまとめてよい:
- 同一画面・同一機能領域の修正
- 受け入れ条件をまとめて書いても粒度が崩れない
- レビュー時間が `size/M` レンジに収まる

### 既に類似 Issue があり、優先度を上げてほしい依頼
- 起票ではなく `gh issue edit <番号> --add-label P0` 等で対応
- 「優先度を P2 → P0 に変更しました」と報告

### スクショ・添付ファイルがある
- GitHub の `gh issue create` は添付に対応しないので、起票後に **Issue 編集 UI で手動添付** をユーザーに依頼する
- 本文には添付予定のファイル名と用途を 1 行入れておく(例: `スクショ: error-toast.png (再現時の画面)`)

### Issue 運用していないリポジトリ
- 「このリポジトリは Issue 運用していますか?」と確認
- 否の場合はこのスキルは適用外。`github-issue-pr-flow` のエッジケース節へ案内

### CLAUDE.md / プロジェクト固有のルールがある
- リポジトリ直下に `CLAUDE.md` があれば、Issue 起票前に必ず一読(プロジェクト固有のラベル運用・テンプレート指定がある可能性)

---

## やってはいけないこと

🚫 As-Is / To-Be が不明確なまま起票する(必ずユーザー確認)
🚫 重複チェックなしで起票する
🚫 起票しただけで実装まで進める
🚫 ユーザー承認なしに新規ラベルを作る
🚫 自分でブランチを切る・PR を作る(実装フェーズは `github-issue-pr-flow` の責任)
🚫 重複候補があっても「念のため新規起票」を独断で選ぶ

---

## チェックリスト(作業終了時)

- [ ] As-Is / To-Be がユーザーに確認済み
- [ ] 該当コード箇所を軽く確認した(または「コードリポジトリではない / 特定不可」を明記)
- [ ] Issue サイズを見積もった(本文「想定サイズ」に記載 / `size/XL` なら分割提案済み)
- [ ] 既存 Issue の重複チェックを実施した
- [ ] ユーザーの選択(a/b/c/d)に従った
- [ ] 種別ラベルを 1 つ付けた(`gh issue view --json labels` で確認済み)
- [ ] サイズラベル(`size/S` / `size/M` / `size/L` / `size/XL`)を 1 つ付けた
- [ ] Issue URL をユーザーに返した
- [ ] 「実装は `github-issue-pr-flow` に引き継ぎ」と案内した
- [ ] ブランチを切っていない・PR を作っていない

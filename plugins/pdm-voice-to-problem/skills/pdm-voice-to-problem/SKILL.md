# PdM｜ユーザーの声 → 問題定義 スキル

CS チケット・問い合わせ・ヒアリングメモなどの**生の声**を、Design Doc に持っていける**問題定義サマリー**に変換する。

> **PdM-Playbook 上の位置**: Skill 1。次のステップは `pdm-design-doc`（Skill 2-3）。

---

## まず確認すること

1. **声の出所** — CS チケット / アプリ内フィードバック / インタビュー / Sales 報告 など
2. **対象期間** — 直近1ヶ月 / 四半期 / リリース後など
3. **件数** — 20件以上ならパターン分類を先に行う
4. **一次情報か二次情報か** — CS チケットは一次（ユーザー本人）、Sales 経由は二次（解釈が混ざる）

期間・件数が曖昧なまま進めると「何件中の何件」が言えず、優先順位付けで崩れる。

---

## 5ステップの実行手順

### Step 1｜前処理

- 期間・件数・出所を確認し、サマリーの先頭にメタ情報として記録
- 20件超なら、まずサンプルを眺めてパターン感をつかむ
- 重複・テンプレ的回答（例: コピペされた問い合わせ）は1件にまとめる

### Step 2｜4分類

声を以下の4つに分ける。④は別扱いにする：

| 分類 | 定義 | 例 |
|---|---|---|
| ① 要望 | 「〇〇してほしい」「〇〇が欲しい」 | 「CSV出力をもっと速くして」 |
| ② 不満 | 「〇〇が使いにくい / つらい / 困る」 | 「画面が重くてイライラする」 |
| ③ 混乱 | 「どこを押せばいいかわからない」「期待と違う動き」 | 「保存ボタンが見つからない」 |
| ④ 報告 | バグ・障害・不正な挙動の報告 | 「決済画面でエラーが出る」 |

**④ 報告は問題定義の対象外**。バグトラッキングへ回す。①②③だけが本スキルの対象。

### Step 3｜掘り下げ（要望は必ず変換する）

声をそのまま問題定義にしない。とくに ① 要望は **「なぜそれを求めるか？」** を3回問い、本当の問題に変換する。

| ユーザーの声 | 掘り下げの問い | 本当の問題 |
|---|---|---|
| 「CSV出力を速くして」 | 遅いと何が困る？ | 進捗の不透明さによる不安 |
| 「ダークモードが欲しい」 | 明るいと何が困る？ | 夜間利用時の眼精疲労 |
| 「一括編集機能が欲しい」 | 1つずつだと何が困る？ | 月末作業に5時間かかる |

掘り下げの代表的な問い：

- **回避方法を聞く**: 「今この問題をどう回避していますか？」 → workaround が見えると痛みの強さがわかる
- **頻度を聞く**: 「週/月に何回くらい起きていますか？」 → 件数だけでは見えない実害
- **代替手段の有無**: 「他にこの目的を達成できる方法はありますか？」 → 解決策に縛られない問題定義に変換

### Step 4｜統合

似た「本当の問題」をパターンにまとめ、件数を集計する：

- 同じ問題を語っているのに表現が違うものは1パターンにまとめる
- パターンごとに件数・代表的な声・深刻度を記録
- パターン間の依存関係（A が解決しないと B が見えない 等）にも気づいたらメモする

### Step 5｜出力

以下のフォーマットで問題定義サマリーを生成する：

```markdown
# ユーザーの声 整理サマリー

- **分析対象**: 〇〇からの声（CSチケット / インタビュー など）
- **期間**: YYYY-MM-DD 〜 YYYY-MM-DD
- **件数**: N件（うち報告系 X件は対象外）
- **分析日**: YYYY-MM-DD

---

## 発見された問題パターン

### 1. [パターン名]（件数: N / N件中）
- **本当の問題**: （Step 3 で掘り下げた結果）
- **代表的な声**:
  - 「〇〇が遅くて困る」（チケット #1234）
  - 「〇〇のとき不安になる」（インタビュー A さん）
- **深刻度**: 高 / 中 / 低（業務への影響度）
- **次のアクション候補**: Design Doc に進む / 追加ヒアリング / 既存機能で対応可

### 2. [パターン名]（件数: N）
...

---

## スコープ外

- バグ・障害報告（X件）→ バグトラッキングへ
- 単発・例外的な声（Y件）→ 後追い候補として保持

---

## Skill 2 への引き継ぎメモ

- **最優先パターン**: パターン #1（理由: 件数 + 深刻度の積で最大）
- **次に検討**: パターン #2
- **要追加調査**: 〇〇の頻度を CS データで確認したい
```

---

## よくある失敗パターン

| 失敗 | 対処 |
|---|---|
| 声をそのまま問題に転記する | 必ず Step 3 の掘り下げを通す |
| 件数だけで優先順位を決める | 深刻度（業務影響）を必ず併記する |
| 一次情報と二次情報を区別しない | 出所をパターンごとに残す（Sales 経由の声 vs CS 直接） |
| 「全員が」「みんな」と一般化する | 分母（N件中 X件）を必ず書く |
| バグ報告を問題定義に混ぜる | ④ 報告は別フローに切り出す |

---

## 重要な原則

- **声 ≠ 問題**: ユーザーは解決策を提案しがち。背後の問題を必ず取り出す
- **件数 ≠ 優先度**: 1件でも事業継続に響く問題はある
- **このスキルの出口**: `pdm-design-doc` に渡せる「問題定義サマリー」。Design Doc を書くのは次のスキル
- **PdM が判断するもの**: どのパターンを取り上げるか、深刻度の評価。AI には組織事情がわからない

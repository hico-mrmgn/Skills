# GitHub Issue → Branch → PR ワークフロー (実装フェーズ)

このスキルは、**既に存在する Issue を実装する** ためのワークフロー。Issue を読み込んでブランチを切り、実装して PR を出すまでを担当する。

**main への直接コミット・直接マージは禁止。マージ実行は必ずオーナー(Tomo)が行う。**

---

## 上位スキルとの関係

```
issue-intake            (As-Is/To-Be → Issue 起票)
       ↓
github-issue-pr-flow    ← このスキル (Issue → ブランチ → 実装 → PR)
```

**Issue がまだ起票されていない場合はこのスキルでは扱わない**。`issue-intake` を呼んで Issue 起票を済ませてから戻ってくる。

---

## このスキルを使う場面

✅ **使う**:
- ユーザーが Issue 番号を提示(「#42 やって」「Issue 12 実装して」)
- バックログから取る指示(「次の P1 やって」「バックログから取って」)
- 「○○を直して」と言われて Issue を確認したら該当 Issue が存在した場合
- ドキュメント・マイグレーション・seed 等、コミットを伴う変更(Issue 経由なら何でも)

❌ **使わない**:
- Issue がまだ起票されていない要望 → `issue-intake` へ
- アイデア段階で仕様を整理したい → `pdm-design-doc` へ
- 読み取り・調査・質問への回答のみ
- ローカル一時ファイル (`/tmp` 等) の操作
- Claude 設定 (`~/.claude/...`) の更新

---

## 標準フロー

### Step 0. 現在のブランチを確認

```bash
git status
git branch --show-current
```

- `main` 以外のブランチに居る場合は、その作業を続けるか新規ブランチを切るかユーザーに確認
- 未コミットの変更がある場合は、何の作業途中か必ず確認してから進める

---

### Step 1. 対象 Issue を確定する

#### パターン A: ユーザーが Issue 番号を明示
そのまま使う。`gh issue view <番号>` で読み込む。

#### パターン B: 「次のやって」「バックログから」
候補を絞り込んで提示する:

```bash
# 優先度ラベルでソート
gh issue list --state open --label "P0" --limit 5
gh issue list --state open --label "P1" --limit 5

# 種別 + 直近作成順
gh issue list --state open --label "bug" --limit 10

# Assignee が自分のもの
gh issue list --state open --assignee @me --limit 10
```

候補を以下のフォーマットでユーザーに提示:

```
バックログから候補(open):

🔴 P0 (0 件)

🟠 P1 (2 件):
- #42 ログイン後の next パラメータ検証で Open Redirect を防止 (created: 2026-04-01)
- #38 課題提出ファイルの MIME 検証強化 (created: 2026-03-28)

🟡 P2 (5 件):
- #45 ...

どれを実装しますか? (番号で指定)
```

ユーザーが選ぶまで実装に入らない。

#### パターン C: 「○○を直して」(Issue 番号無し)
1. `gh issue list --search "<キーワード>" --state open --limit 10` で既存 Issue を探す
2. 該当があれば「`#X` のことですか?」と確認
3. 該当が無ければ **`issue-intake` を呼んで先に起票するようユーザーに案内**(このスキルでは起票しない)

---

### Step 2. Issue を読み込み、スコープを確認

```bash
gh issue view <番号> --json title,body,labels,assignees,milestone,state
```

確認するポイント:

- **タイトル・本文(As-Is / To-Be / 受け入れ条件)** が今回やるべきこととズレていないか
- **ラベル**(種別・優先度・領域) — どの程度の規模感の作業か
- **State** が open か(closed なら再オープンが必要か確認)
- **古い Issue でないか** — 数ヶ月前の Issue は仕様が変わっている可能性あり。1 文で要約を返してユーザーに「これでいい?」と確認する
- **関連 Issue / PR** — 本文内の `#X` リンクや、近いブランチ名の存在を軽く確認

**スコープが Issue と乖離していたらユーザーに確認**:
- 「Issue には A だけ書いてありますが、合わせて B もやりますか?」
- 「Issue から見て A は別 Issue に分けたほうが良さそうですが、どうしますか?」

承認なしにスコープを広げない。

---

### Step 3. Issue 番号入りのブランチを切る

```bash
git checkout main
git pull
git checkout -b <番号>-<短い英語ケバブケース説明>
```

例: Issue #42 が「ログイン画面の文言修正」なら → `42-fix-login-copy`

**ルール**:
- ブランチ名の先頭は Issue 番号
- 区切りは `-`
- 説明は英語・小文字・ケバブケース・5 単語以内
- リポジトリで既に `feature/` `fix/` などプレフィックスを使う運用がある場合はそちらに合わせる(`git branch -a` で確認)

**注意**: パターン B/C で複数 Issue を渡された場合、原則 **1 Issue = 1 ブランチ = 1 PR**。まとめたい場合はユーザーに確認。

---

### Step 4. 実装・コミット

- 通常の編集ルールに従って実装
- コミットメッセージは既存スタイルに合わせる(`git log --oneline -20` で確認)
- 1 つの PR に詰め込みすぎない。スコープが膨らんだら別 Issue に分ける(その場合は `issue-intake` で追加 Issue を起票)
- プロジェクトの `CLAUDE.md` がある場合は必ず参照(型チェック・lint・ビルドの完了確認など)

#### Step 4-b. E2E テストの追加(必須)

**機能追加・画面追加・UI 変更・業務フロー変更**を行った場合は、**同じ PR 内で対応する E2E テストも追加する**。テストを別 PR に分けない。

**手順**:

1. **プロジェクトに E2E 環境があるか確認**
   - `package.json` / `playwright.config.*` / `cypress.config.*` / `e2e/` ディレクトリ等を確認
   - 無い場合 → E2E 追加義務はスキップ(ただしユーザーに「E2E 環境が無いためテスト未追加」と必ず報告)

2. **既存テストで今回の変更がカバーされているか確認**
   - 影響範囲のページ・コンポーネントに対応する `*.spec.ts` を `grep` / ファイル名検索で探す
   - 既存テストがあり、変更によって挙動が変わる → そのテストを更新
   - 既存テストはあるが今回の追加箇所は未カバー → ケース追加
   - そもそも対応テストが無い → 新規 `*.spec.ts` を作成

3. **最低限カバーするケース**
   - ハッピーパス(正常系で完走できる)
   - 主要なバリデーションエラー or 権限エラー or 空状態 のいずれか 1 ケース以上
   - ロール別アクセス制御がある場合 → ロール切り替えのケース

4. **テスト方針**
   - プロジェクトに `playwright-test` 等のスキルがあれば**必ず参照**
   - ロケーターは `getByRole` / `getByLabel` 優先。`data-testid` は最終手段
   - 既存の Page Object Model (POM) が使えるなら再利用

5. **テスト実行**
   - 追加・修正したテストをローカルで実行し、グリーンであることを確認
   - 関連する既存テストも回し、リグレッションが無いことを確認

**E2E を追加しなくてよいケース**(該当する場合のみ):
- 純粋なリファクタで挙動・UI に一切変更が無い
- ドキュメントのみの変更
- ビルド・CI・依存関係のメンテナンスで挙動に影響しない
- バックエンド内部実装の差し替えで、エンドツーエンドの観測可能挙動が変わらない(Vitest 等の単体テストでカバー)

**E2E を追加できないが追加すべきケース**(時間・技術的制約等):
PR 本文に「E2E テスト未追加(理由: ...)」と必ず明記し、フォローアップ Issue を `issue-intake` で起票。ユーザーに口頭でも報告。サイレントに飛ばさない。

---

### Step 5. push して PR を作成

```bash
git push -u origin HEAD
gh pr create --title "<簡潔なタイトル>" --body "$(cat <<'EOF'
## Summary
- <1-3 行の変更概要>

## Changes
- <変更点1>
- <変更点2>

## Test plan
- [ ] <検証項目1>
- [ ] <検証項目2>

Closes #<Issue番号>

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

**重要**:
- 本文に必ず `Closes #<Issue番号>` を入れる(後述の自動クローズに必須)
- PR タイトルは Issue タイトルと揃えるか、より具体化する
- `--draft` を付けるかは状況次第。CI を回しつつ続きの作業がある場合はドラフトにする

#### Issue 自動クローズの仕組み(必読)

GitHub は PR 本文または各コミットメッセージに次のキーワードがあると、PR が **デフォルトブランチ(= main)にマージされたタイミングで参照先 Issue を自動クローズ** する。

- `Closes #<番号>` / `Close #<番号>` / `Closed #<番号>`
- `Fixes #<番号>` / `Fix #<番号>` / `Fixed #<番号>`
- `Resolves #<番号>` / `Resolve #<番号>` / `Resolved #<番号>`

**ルール**:
- このスキルでは `Closes #<番号>` で統一する(英語動詞の揺れを作らない)
- PR が複数 Issue を閉じる場合は `Closes #12, Closes #34` のように **各 Issue に対してキーワードを繰り返す**(`Closes #12, #34` は動作しない)
- 別リポジトリの Issue を閉じたい場合は `Closes owner/repo#番号` 形式
- PR がドラフトでもマージ時に自動クローズされる(マージ条件のみが効く)
- Squash / Rebase / Merge いずれでも動作する
- 自動クローズが**効かない条件**:
  - PR が main 以外のブランチにマージされた場合
  - キーワードが PR タイトル / レビューコメント / コードコメントにしかない場合(PR 本文かコミット本文に書く)
  - キーワードがコードブロック・引用内にある場合

**確認手順**: PR 作成後に `gh pr view <PR番号>` で本文に `Closes #<番号>` が入っていることを目視で確認する。入っていなければ `gh pr edit <PR番号> --body "..."` で修正する。

---

### Step 6. 完了報告

ユーザーに以下を報告して **作業を止める**:

- Issue URL
- PR URL
- 変更ファイル一覧(箇条書き)
- 触らなかった範囲(意図して触らなかったもの)
- マニュアル/テスト更新の有無(プロジェクトの CLAUDE.md で必須化されていれば必ず明記)

**やってはいけないこと**:

🚫 `gh pr merge` を実行しない
🚫 `git push origin main` などで main に直接 push しない
🚫 `--admin` フラグで PR を強制マージしない
🚫 ユーザーに無断で main にリベース・force push しない
🚫 **`gh issue close` を実行しない**(後述)

ユーザーが「マージして」「merge して」と**明示的**に指示した場合のみマージを実行する。それ以外はマージしない。

---

## Issue クローズのタイミング(厳守)

**Issue は PR が main にマージされた瞬間に、GitHub が `Closes #N` キーワードを検知して自動でクローズする。Claude が手動で閉じることは禁止。**

🚫 **禁止する操作**:
- `gh issue close <番号>` を実行する
- GitHub Web UI で "Close issue" ボタンを押すよう促す
- 「実装が終わったので Issue を閉じておきますね」と先回りしてクローズする
- PR を作成した時点で Issue を閉じる(マージ前のクローズ)
- PR をマージせずに別経路(直接コミット・cherry-pick 等)で対応した場合に、Issue だけ閉じる

✅ **正しい挙動**:
- PR 作成後、Issue は **OPEN のまま放置** する
- ユーザー(Tomo)が main にマージする
- GitHub が自動で Issue をクローズする(`closedByPullRequestsReferences` に PR が記録される)
- Claude 側からは Issue クローズに一切関与しない

**例外**:
ユーザーが**明示的に**「Issue #N を閉じて」と指示した場合のみ `gh issue close` を実行してよい。「実装終わった」「PR 作った」「マージ済み」だけでは閉じる根拠にならない — その場合も自動クローズに任せる。

**Issue を間違って実装外の理由でクローズしたい場合**(例: 重複・仕様変更で不要になった):
これも Claude からは閉じない。ユーザーに「この Issue は不要になったようです。閉じますか?」と確認し、明示の "閉じて" を待つ。

## エッジケース

### Issue がまだ起票されていない
このスキルでは起票しない。ユーザーに **「`issue-intake` を先に呼んで Issue を起票してから戻ってきてください」** と案内する。Claude 側から勝手に `gh issue create` を叩かない。

### Issue ベースで運用していないリポジトリの場合
ユーザーに「このリポジトリは Issue 運用していますか?」と確認する。否であれば Issue ステップを飛ばし、ブランチ + PR のみで進める(タイトル・本文には変更内容を直接書く)。

### 緊急修正でフローを省略したい場合
ユーザーから「直接 main に push して」「Issue 無しで PR だけ作って」など明示の指示があった場合のみ、その指示に従う。Claude 側から省略を提案しない。

### 既存のブランチで作業中の場合
途中のブランチがある場合は、続行か別ブランチを切るかをユーザーに確認。勝手に新規ブランチに切り替えない。

### 実装中にスコープが膨らんだ
- 元 Issue の範囲を超える変更が必要になったら、**実装を止めてユーザーに確認**
- 「追加分は別 Issue に分けますか? それとも元 Issue を更新して取り込みますか?」
- 別 Issue にする場合は `issue-intake` で起票してから別ブランチで対応

### Issue 本文と実態が違う(古い・曖昧)
- Issue 本文に書いてある As-Is / To-Be が現状と合わない場合、**実装前に Issue 自体を更新するべきかユーザーに確認**
- 黙って本文を書き換えない

## チェックリスト(作業終了時)

- [ ] 対象 Issue を確定し、スコープをユーザーと合意した
- [ ] Issue 番号入りのブランチを切った
- [ ] コミットメッセージはリポジトリのスタイルに沿っている
- [ ] **E2E テストの追加・更新を行った**(または「不要 / 環境無し / 未追加+理由」のいずれかを PR 本文に明記)
- [ ] 追加・更新した E2E テストがローカルでグリーン
- [ ] PR 本文に `Closes #<番号>` が入っている(`gh pr view` で目視確認済み)
- [ ] main へのマージは**していない**
- [ ] **Issue は手動でクローズしていない**(OPEN のまま PR マージ待ち)
- [ ] ユーザーに Issue URL と PR URL を報告した
- [ ] 「main へマージされれば Issue は自動クローズされる」旨をユーザーに伝えた
